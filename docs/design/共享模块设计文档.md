# 共享模块设计文档

## 一、概述

Kunlun-M 的扫描引擎采用**核心框架 + 语言引擎插件**的架构。核心框架位于 `core/core_engine/` 目录，提供所有语言引擎共享的抽象层和数据结构。各语言引擎（php/javascript/python/java/go/c）位于同目录下的子文件夹中，每个引擎只需实现特定语言的解析器和回溯逻辑，即可接入统一的扫描流水线。

本文档深入分析 4 个核心共享模块的设计原理、数据结构和在回溯分析中的具体作用：

| 模块 | 文件 | 行数 | 职责 |
|------|------|------|------|
| 函数摘要 | `function_summary.py` | 196 | 跨文件函数摘要生成、缓存、查询 |
| 分支约束 | `branch_constraint.py` | 52 | 分支条件约束的统一数据结构 |
| 追踪缓存 | `trace_cache.py` | 108 | 运行时变量可控性缓存 + 内置知识库桥接 |
| 内置知识库 | `builtin_knowledge.py` | 1189 | 语言内置函数的返回值可控性知识 |

```
core/core_engine/
├── __init__.py
├── function_summary.py    ← 函数摘要 + 磁盘缓存
├── branch_constraint.py   ← 分支约束数据结构
├── trace_cache.py         ← 运行时缓存 + 知识库桥接
├── builtin_knowledge.py   ← 4语言内置函数知识库
├── php/
│   └── parser.py           ← PHP 引擎（含语言专属 builtin_knowledge.py）
├── javascript/
│   └── parser.py
├── python/
│   └── parser.py
├── java/
│   └── parser.py
├── go/
│   ├── parser.py
│   └── _ast_trace.py
└── c/
    ├── parser.py
    └── builtin_knowledge.py  ← C/C++ 专属知识库
```

## 二、函数摘要模块 (`function_summary.py`)

### 2.1 设计动机

静态分析中最大的性能瓶颈之一是**跨文件追踪**。当回溯分析遇到一个用户自定义函数调用时，必须进入该函数体分析其返回值是否可控。如果每次遇到调用都重新解析和遍历函数体，复杂度将呈指数增长。

函数摘要机制的核心思想：**对每个函数进行一次性分析，生成"返回值依赖摘要"，之后通过查摘要替代重新分析**。

### 2.2 核心数据结构

#### 2.2.1 ReturnFlowItem — 单条返回值数据流

```python
@dataclass
class ReturnFlowItem:
    """单条返回值数据流"""
    order: int              # 该 return 语句在函数中的出现顺序（0, 1, 2...）
    return_index: int        # return 语句在源码中的行号
    origin: str              # 返回值的来源描述（变量名、表达式文本等）
    origin_type: str         # 来源类型标识
    dep_params: List[int]    # 返回值依赖的参数索引列表（0-indexed）
    path: List[Dict]         # 返回值传播路径（经过的中间变量和操作）
```

**关键设计点**：

- **order** 字段用于区分同一函数中多个 return 语句。引擎在分析时会将每个 return 语句独立处理，生成各自的 `ReturnFlowItem`。调用方可以遍历所有 order 来判断是否存在一条可控的返回路径。
- **dep_params** 是整个摘要系统的核心字段。它记录了"这个 return 语句的返回值依赖于函数的哪些参数"。例如 `return param1 + param2` 则 `dep_params = [0, 1]`，`return "hardcoded"` 则 `dep_params = []`。
- **path** 记录数据在函数体内的传播过程，例如 `param1 → tmp → filter(tmp) → return`，每一步是一个 `{var, line, op}` 字典。这个路径信息主要用于调试和漏洞链展示，不直接影响判定逻辑。

#### 2.2.2 FunctionSummary — 单个函数摘要

```python
@dataclass
class FunctionSummary:
    """单个函数的完整摘要"""
    name: str                              # 函数名
    params: List[str]                       # 参数名列表
    line_range: Tuple[int, int]            # 函数体行号范围 [start, end]
    return_flow: List[ReturnFlowItem]      # 所有 return 语句的数据流
    receiver_name: str = ""               # 接收者名（Go 的方法接收者）
    is_method: bool = False                # 是否为方法（Go 引擎使用）
```

**关键设计点**：

- **receiver_name** 和 **is_method** 是为 Go 引擎特别添加的字段。Go 语言的函数和方法有区别——方法有接收者（receiver），调用方式不同。这两个字段让摘要系统能区分 `func foo()` 和 `func (s *Struct) foo()`。
- **line_range** 在判断函数调用是否落入已知函数体时使用，避免误匹配同名函数。

#### 2.2.3 FileSummary — 单文件摘要

```python
@dataclass
class FileSummary:
    """单文件摘要"""
    file: str                              # 文件路径
    content_hash: str                       # 文件内容 SHA256 哈希
    functions: List[FunctionSummary]      # 文件中所有函数的摘要
```

#### 2.2.4 数据序列化

所有三个数据类都实现了 `to_dict()` 和 `from_dict()` 方法，支持完整的 JSON 序列化/反序列化。这为磁盘缓存提供了基础——摘要可以持久化存储，跨扫描复用。

### 2.3 摘要缓存管理器 (`SummaryCacheManager`)

#### 2.3.1 缓存策略

```
.kunlun_cache/
├── index.json           ← {file_path: content_hash} 映射表
└── summaries/
    ├── a1b2c3d4e5f6g7h8.json   ← 文件摘要（文件名 = file_path 的 SHA256 前16位）
    └── ...
```

缓存策略基于**内容哈希比对**：

1. 对每个源文件计算 SHA256 哈希
2. 与 `index.json` 中记录的历史哈希对比
3. **哈希相同 → 从 summaries/ 加载缓存**（缓存命中）
4. **哈希不同或文件不存在 → 标记需要重新生成**（返回空的 functions 列表）

这个设计确保：
- 文件未修改时不会重复分析（大幅减少扫描时间）
- 文件修改后自动失效旧缓存
- 缓存粒度为文件级，一个文件的任何修改只影响该文件的摘要

#### 2.3.2 关键方法

```python
class SummaryCacheManager:
    def __init__(self, cache_dir=".kunlun_cache")
    
    def load_or_generate(self, target_path, files_dict) -> Dict[str, FileSummary]:
        """
        批量加载或标记需要重新生成的文件摘要。
        返回值中 functions 为空表示需要重新生成。
        """
    
    def save_file_summary(self, target_path, file_path, summary) -> None:
        """保存单个文件摘要到缓存"""
    
    def clear(self, target_path) -> None:
        """删除整个缓存目录"""
```

**`load_or_generate` 的返回语义**是整个缓存系统最微妙的设计：

- 返回的 `FileSummary.functions` 不为空 → 引擎可以直接使用，无需再分析
- 返回的 `FileSummary.functions` 为空 → 引擎需要解析该文件并生成摘要，然后调用 `save_file_summary` 保存

这种"标记而非自动生成"的设计将**缓存管理和摘要生成分离**，让引擎可以按自己的节奏生成摘要（有的引擎可能先扫描 sink 再按需生成摘要）。

### 2.4 在回溯分析中的角色

函数摘要在回溯分析中的使用流程：

```
回溯遇到函数调用 func(arg)
    │
    ├─ 1. 查 TraceCache.get() — 运行时缓存是否已有结果？
    │   └─ 命中 → 直接使用
    │
    ├─ 2. 查 BuiltinKnowledge.lookup() — 是否为内置函数？
    │   ├─ safe=True → 返回 code=-1（安全）
    │   ├─ passthrough=[n] → 返回 ('deps', [dep_vars])
    │   └─ 不在知识库 → 继续步骤3
    │
    ├─ 3. 查 FunctionSummary — 是否有该函数的摘要？
    │   ├─ 有摘要 → 遍历 return_flow：
    │   │   ├─ dep_params 包含参数位置 → 继续追踪该参数来源
    │   │   └─ dep_params 为空 → 返回值不可控（硬编码），阻断
    │   └─ 无摘要 → 进入函数体分析（递归回溯）
    │
    └─ 4. 分析完成后写入 TraceCache.put() — 缓存结果
```

**典型场景**：分析 `echo escape(user_input)` 时：
1. 回溯到 `escape(user_input)`
2. 查内置知识库 → `escape` safe=True → 返回 code=-1（安全，漏洞被过滤）
3. 不需要进入 `escape` 函数体

**跨文件场景**：分析 `echo get_data($_GET['id'])` 时：
1. 回溯到 `get_data($_GET['id'])`
2. 不是内置函数
3. 查函数摘要 → 假设 `get_data` 的摘要中有一条 return_flow: `dep_params=[0]`, `origin="$_GET['id']"`
4. 追踪参数 0 → 确认来自 `$_GET['id']`（可控）
5. 漏洞成立

## 三、分支约束模块 (`branch_constraint.py`)

### 3.1 设计动机

考虑以下 PHP 代码：

```php
if ($role == 'admin') {
    echo $query;  // 只有 admin 才能到达这里
} else {
    echo "Access denied";
}
```

如果不考虑 if 条件，分析器会认为 `$query` 在 echo 处可能可控，报告一个误报。但实际上，只有 `$role == 'admin'` 时才能进入这个分支，而 `$role` 的值可能通过其他方式固定，使得这条路径不可达。

分支约束追踪的核心思想：**在回溯过程中收集经过的条件约束，判断这些约束是否与数据流矛盾**。

### 3.2 BranchConstraint 数据结构

```python
class BranchConstraint:
    """单个条件约束"""
    __slots__ = ('var_name', 'op', 'value')
    
    def __init__(self, var_name: str, op: str, value=None):
        self.var_name = var_name    # 约束变量名
        self.op = op                # 操作符：==, !=, isset, !isset, in, not in
        self.value = value          # 约束值（可选）
```

**使用 `__slots__` 优化内存**：BranchConstraint 在回溯过程中会被大量创建（每个 if/else 分支、每个循环条件都会产生约束）。使用 `__slots__` 避免了每个实例的 `__dict__` 开销，在大型代码库中显著减少内存占用。

**约束的三元组语义**：`(var_name, op, value)` 构成一个完整的逻辑命题：
- `("role", "==", "admin")` → role 等于 admin
- `("input", "!=", "")` → input 不等于空字符串
- `("data", "isset", None)` → data 已设置（PHP 特有）

### 3.3 取反操作 (`negate`)

```python
def negate(self):
    """取反约束，用于 else/else if 分支"""
    neg_map = {
        '==': '!=', '===': '!==',
        '!=': '==', '!==': '===',
        'isset': '!isset', '!isset': 'isset',
        'in': 'not in', 'not in': 'in',
    }
    return BranchConstraint(
        var_name=self.var_name,
        op=neg_map.get(self.op, self.op),
        value=self.value,
    )
```

**设计原理**：当进入 if/else 分支时，if 的条件约束需要取反才能代表 else 分支的约束。

```python
if ($x == 1):        # 约束: BranchConstraint("x", "==", 1)
    SINK($data)       # 在此分支中，x 必定等于 1
else:
    SINK($data)       # 在此分支中，x 必定不等于 1（取反）
```

**支持的取反映射**：
| 原操作符 | 取反后 | 说明 |
|---------|--------|------|
| `==` | `!=` | 等值判断 |
| `===` | `!==` | 严格等值（JS/PHP） |
| `isset` | `!isset` | PHP 变量存在性 |
| `in` | `not in` | 值集合判断（C 引擎特有） |

### 3.4 约束冲突检测（各引擎自行实现）

`BranchConstraint` 只提供数据结构，约束冲突检测逻辑在**各语言引擎中独立实现**。这是因为不同语言的分支语法差异很大：

| 引擎 | 分支语法 | 检测函数 |
|------|---------|---------|
| PHP | if/elseif/else, 三元, match | `_check_sink_branch_constraints`, `_check_if_branch_constraint` |
| JavaScript | if/else, 三元, switch | `_check_branch_constraints`, `_check_switch_constraints` |
| Python | if/elif/else, 三元, match-case | `_check_branch_constraints`, `_check_match_constraints` |
| Java | if/else, 三元, switch | `_check_branch_constraints`, `_check_switch_constraints` |
| Go | if/else, switch, for | `_check_branch_constraints` |
| C/C++ | if/else, 三元, switch, while/do-while | `_check_sink_branch_constraints`, `_check_if_branch_constraint`, `_check_while_constraint` |

**约束冲突的判定逻辑**（以 C 引擎为例）：

```python
# 正在追踪的变量 source_var = "input"
# 分支约束: BranchConstraint("input", "==", "safe_value")

# 如果 input == "safe_value"（固定值），则 input 不可控
# → 阻断回溯，认为此路径不可能携带用户输入
```

### 3.5 约束传播机制

约束在回溯过程中的传播方式：

```
SINK(var)  ← 在 if 分支内
    │
    │  branches_constraints = [
    │      BranchConstraint("x", "==", "fixed"),
    │      BranchConstraint("role", "!=", "admin")
    │  ]
    │
    ▼
parameters_back(var, ..., branches_constraints)
    │
    │  每遇到新的 if/else：
    │  - 提取当前条件为 constraint
    │  - if 体：追加 constraint 到列表
    │  - else 体：追加 constraint.negate() 到列表
    │  - 退出分支后：移除该约束
    │
    ▼
```

约束列表随递归深度变化，像一个**栈**：进入分支时压入约束，离开分支时弹出约束。这种设计确保了每个变量追踪点处都拥有正确的约束上下文。

## 四、追踪缓存模块 (`trace_cache.py`)

### 4.1 设计动机

回溯分析存在大量**重复计算**：

1. 同一变量在同一行可能被多次追踪（不同 sink 指向同一变量）
2. 同一函数的摘要可能被多次查询
3. 复杂表达式的追踪路径可能重复

`TraceCache` 提供两层缓存来解决这些问题：
- **运行时缓存**：缓存已分析的变量可控性结果
- **内置知识库桥接**：通过统一接口查询语言内置函数知识

### 4.2 TraceCache 类设计

```python
class TraceCache:
    """变量追踪结果缓存，含内置知识库预载"""
    
    def __init__(self, language):
        self.language = language                    # 语言标识
        self._runtime_cache = {}                    # 运行时缓存
        self._builtin_module = None                 # 延迟加载的知识库模块
```

### 4.3 运行时缓存

#### 4.3.1 缓存键设计

```python
def _make_key(self, file_path, var_name, lineno):
    return (file_path, str(var_name), int(lineno))
```

**三元组键 `(file_path, var_name, lineno)`** 的设计含义：

- **file_path**：同一变量名在不同文件中含义不同（不同文件的局部变量）
- **var_name**：要追踪的变量名
- **lineno**：变量在源码中的行号。同一变量在不同行可能有不同的值（经过赋值后值改变）

这个三元组唯一标识了一个"变量在某文件某行的值"，是追踪结果的最细粒度缓存单元。

#### 4.3.2 缓存值设计

```python
def get(self, file_path, var_name, lineno):
    """返回 (code, cp, expr_lineno) 或 None"""
    
def put(self, file_path, var_name, lineno, result):
    """result = (code, cp, expr_lineno)"""
```

缓存值是引擎统一的**追踪结果三元组**：

| 字段 | 含义 | 典型值 |
|------|------|--------|
| `code` | 可控性状态码 | `0`（可控）, `-1`（不可控/安全）, `1`（需继续追踪） |
| `cp` | 控制参数/变量信息 | `param1`, `$_GET['id']`, `req.body` |
| `expr_lineno` | 表达式所在行号 | 整数 |

#### 4.3.3 生命周期管理

```python
def clear(self):
    """清空运行时缓存（内置知识库不受影响）"""
    self._runtime_cache.clear()

def invalidate(self, file_path, var_name, lineno):
    """删除指定缓存条目"""
```

- `clear()` 在**每次新扫描开始时调用**，确保不污染不同扫描任务的结果
- `invalidate()` 用于精细化的缓存失效（如发现新的信息需要更新之前的判断）

### 4.4 内置知识库桥接

#### 4.4.1 延迟加载策略

```python
def _load_builtin_module(self):
    """延迟加载语言知识库模块"""
    if self._builtin_module is None:
        try:
            module_path = f"core.core_engine.{self.language}.builtin_knowledge"
            self._builtin_module = importlib.import_module(module_path)
        except ImportError:
            self._builtin_module = False
    return self._builtin_module
```

**延迟加载的设计原因**：
1. 扫描可能只涉及一种语言，不需要加载其他语言的知识库
2. Go 和 C 引擎的内置知识库是独立的文件（`go/builtin_knowledge.py` 和 `c/builtin_knowledge.py`），而 PHP/JS/Python/Java 共享顶层 `builtin_knowledge.py`
3. 知识库模块加载后是常量数据，不会变化，所以只加载一次

#### 4.4.2 统一查询接口

```python
def lookup_builtin(self, func_name):
    """
    查询内置知识库
    :param func_name: 函数/方法名（支持 "module.func" 和 "func" 两种格式）
    :return: {"passthrough": [...], "safe": bool} 或 None
    """
    module = self._load_builtin_module()
    if module and hasattr(module, 'lookup'):
        return module.lookup(func_name)
    return None
```

**这个方法是引擎和知识库之间的唯一桥梁**。引擎不需要知道知识库的具体实现，只需调用 `cache.lookup_builtin(func_name)` 即可。

**查询格式支持两种形式**：
- 完全限定名：`"html.escape"`, `"str.upper"`, `"StringEscapeUtils.escapeHtml4"`
- 短名（方法名）：`"escape"`, `"upper"`, `"trim"`

引擎通常会先尝试完全限定名匹配，失败后再尝试短名匹配。

### 4.5 两层缓存的协作关系

```
引擎调用 cache.lookup_builtin("htmlspecialchars")
    │
    TraceCache
    ├─ _load_builtin_module() → 加载 builtin_knowledge.py
    └─ module.lookup("htmlspecialchars")
        └─ 返回 {"passthrough": [0], "safe": True}
            │
            引擎判断：safe=True → 直接返回 -1（安全，无需继续追踪）
```

```
引擎追踪变量 x，行 42，文件 a.php
    │
    TraceCache.get("a.php", "x", 42)
    ├─ 命中 → 直接返回缓存结果
    └─ 未命中 → 引擎执行追踪 → TraceCache.put("a.php", "x", 42, result)
```

### 4.6 TraceCache 与 FunctionSummary 的关系

两者都提供"避免重复分析"的能力，但层面不同：

| 维度 | TraceCache（运行时缓存） | FunctionSummary（函数摘要） |
|------|------------------------|--------------------------|
| 粒度 | 单个变量在某行某文件 | 整个函数的返回值数据流 |
| 持久性 | 内存级，每次扫描清空 | 磁盘级，文件未修改可复用 |
| 用途 | 避免同一变量的重复追踪 | 避免同一函数的重复分析 |
| 生成时机 | 追踪过程中按需生成 | 扫描前批量生成或首次遇到时按需生成 |

## 五、内置函数知识库 (`builtin_knowledge.py`)

### 5.1 设计动机

静态分析引擎面临一个根本性挑战：**语言提供了大量内置函数，逐一分析其源码既不可能也无必要**。内置函数的行为是确定的、公开的——`strlen()` 永远返回整数，`htmlspecialchars()` 永远做 HTML 转义。

知识库的核心思想：**将内置函数的返回值可控性信息硬编码为静态数据，让引擎直接查表而不需要分析函数体**。

### 5.2 知识条目结构

每个知识条目是一个字典：

```python
{
    "passthrough": [0, 1],  # 返回值依赖的参数位置（0-indexed）
    "safe": False           # 是否做了有效安全过滤
}
```

**`passthrough` 字段的语义**：

- `[0]` → 返回值依赖第 1 个参数（index 0）。如 `str.upper(s)` 返回的值取决于 `s`。
- `[1]` → 返回值依赖第 2 个参数（index 1）。如 `str_replace(pattern, replacement, subject)` 返回的值取决于 `subject`（index 2），但 PHP 引擎中 `str_replace` 的 passthrough 是 `[2]`。
- `[0, 1]` → 返回值依赖前两个参数。如 `zip(a, b)` 的返回值同时取决于 a 和 b。
- `[]` → 返回值与任何参数无关。如 `len(s)` 返回整数，不受输入影响。

**`safe` 字段的语义**：

- `True` → 该函数做了有效安全过滤，即使 `passthrough` 非空，返回值也不再构成安全威胁。
  - 例如 `htmlspecialchars(s)` 的返回值虽然来自 `s`，但已经过 HTML 转义，不会产生 XSS。
- `False` → 该函数未做安全过滤，返回值可能仍然危险。

**这两个字段的组合含义**：

| passthrough | safe | 含义 | 引擎行为 |
|-------------|------|------|---------|
| `[]` | `True` | 返回值与输入无关，安全 | `code=-1`（不可控） |
| `[0]` | `False` | 返回值透传第1个参数，未过滤 | 继续追踪第1个参数 |
| `[0]` | `True` | 返回值来自第1个参数，但已过滤 | `code=-1`（安全） |

### 5.4 查询机制

```python
@classmethod
def lookup(cls, language, func_name):
    """
    查询内置函数知识库
    :param language: "python", "php", "javascript", "java"
    :param func_name: 函数/方法名
    :return: {"passthrough": [...], "safe": bool} or None
    """
```

**两级匹配策略**：

1. **精确匹配**：直接在知识库字典中查找 `func_name`
2. **短名匹配**：如果 `func_name` 包含 `.`（如 `html.escape`），提取最后部分（`escape`）再次查找

```python
# 精确匹配
if func_name in kb:
    return kb[func_name]

# 短名匹配（方法名不带类/模块前缀）
if "." in func_name:
    short_name = func_name.split(".")[-1]
    if short_name in kb:
        return kb[short_name]
```

**短名匹配的设计原因**：

在静态分析中，函数调用链的解析往往不完整——引擎可能只拿到方法名而拿不到完整的类路径。例如：

```python
# 源码中：result = obj.escape(data)
# 引擎只能识别出方法名 "escape"，不知道 obj 的类型
# 短名匹配让引擎仍然能找到 "html.escape" 的知识
```

这种设计在**覆盖广度**和**精确度**之间做了折中：牺牲了部分精确度（可能匹配到错误的类），但大幅提高了知识库的命中率。

### 5.6 知识库在回溯中的具体应用

#### 场景 1：安全过滤函数阻断追踪

```php
// PHP 源码
echo htmlspecialchars($_GET['name']);

// 回溯过程：
// 1. SINK = echo, 参数 = htmlspecialchars($_GET['name'])
// 2. 追踪 htmlspecialchars($_GET['name'])
// 3. 查知识库：htmlspecialchars → {passthrough: [0], safe: true}
// 4. safe=true → code=-1（安全，漏洞不成立）
// 5. 即使 passthrough=[0]，也不继续追踪 $_GET['name']
```

#### 场景 2：透传函数继续追踪

```python
# Python 源码
os.system(user_input.upper())

# 回溯过程：
# 1. SINK = os.system, 参数 = user_input.upper()
# 2. 追踪 user_input.upper()
# 3. 查知识库：upper → {passthrough: [0], safe: false}
# 4. safe=false + passthrough=[0] → 返回值来自参数 0（即 user_input）
# 5. 继续追踪 user_input → 确认可控 → 漏洞成立
```

#### 场景 3：非透传函数阻断追踪

```python
# Python 源码
os.system(str(len(user_input)))

# 回溯过程：
# 1. SINK = os.system, 参数 = str(len(user_input))
# 2. 追踪 str(len(user_input))
# 3. 查知识库：str → {passthrough: [0], safe: false} → 继续追踪 len(user_input)
# 4. 查知识库：len → {passthrough: [], safe: true}
# 5. passthrough=[] → 返回值与参数无关 → 不可控 → 漏洞不成立
```

#### 场景 4：多参数透传

```javascript
// JS 源码
element.innerHTML = Object.assign(defaults, userInput);

// 回溯过程：
// 1. SINK = innerHTML, 参数 = Object.assign(defaults, userInput)
// 2. 查知识库：Object.assign → {passthrough: [0, 1], safe: false}
// 3. passthrough=[0,1] → 返回值同时依赖 defaults 和 userInput
// 4. 对参数 0 (defaults) 和参数 1 (userInput) 分别追踪
// 5. defaults 不可控，userInput 可控 → 漏洞成立
```

### 6.2 各模块的交互时序

以一个跨文件追踪场景为例：

```
文件 A.php:
    function get_query($table, $id) {
        return "SELECT * FROM " . $table . " WHERE id=" . $id;
    }
    
    $query = get_query("users", $_GET['uid']);
    $db->query($query);

回溯分析时序：

1. 规则匹配 SINK: $db->query($query)，提取参数 $query
    
2. 引擎调用 function_back_php("$query", ...)

3. TraceCache.get("A.php", "query", line=4) → 未命中

4. _find_var_definition("$query") → 找到 $query = get_query("users", $_GET['uid'])

5. 遇到函数调用 get_query()
   ├─ TraceCache.lookup_builtin("get_query") → None（不是内置函数）
   └─ 查 FunctionSummary：
       ├─ SummaryCacheManager.load_or_generate() → 缓存未命中
       ├─ 解析 A.php 生成 get_query 的摘要：
       │   return_flow = [
       │       ReturnFlowItem(
       │           order=0,
       │           return_index=3,
       │           origin='"SELECT * FROM " . $table . " WHERE id=" . $id',
       │           origin_type='concat',
       │           dep_params=[0, 1],  // 依赖参数 $table 和 $id
       │           path=[{var: '$table', line: 3}, {var: '$id', line: 3}]
       │       )
       │   ]
       ├─ SummaryCacheManager.save_file_summary() → 保存到磁盘
       └─ 返回摘要

6. 摘要显示 dep_params=[0, 1]，分别追踪参数 0 和 1：
   ├─ 参数 0: $table = "users" → 硬编码字符串 → 不可控
   │   └─ BranchConstraint 检查：无约束冲突 → 但来源是字面量，不可控
   └─ 参数 1: $id = $_GET['uid'] → 可控！

7. dep_params 中有一个可控参数 → 返回值可控

8. $query 可控 → $db->query($query) 存在 SQL 注入

9. TraceCache.put("A.php", "query", line=4, (0, "$_GET['uid']", line))
```

### 6.3 性能优化效果

| 优化机制 | 优化前 | 优化后 | 提升倍数 |
|---------|--------|--------|---------|
| 运行时缓存（TraceCache） | 同一变量多次完整追踪 | 首次追踪后直接查表 | O(n) → O(1) |
| 函数摘要（FunctionSummary） | 每次调用重新分析函数体 | 一次性生成，多次查摘要 | O(n×m) → O(n) + O(m) |
| 磁盘缓存（SummaryCacheManager） | 每次扫描重新解析所有文件 | 文件未修改直接加载 | 首次扫描时间 × 扫描次数 → 首次扫描时间 |
| 内置知识库（BuiltinKnowledge） | 尝试进入内置函数体分析 | 直接查表判断可控性 | O(∞) → O(1) |

## 七、设计哲学总结

### 7.1 分层抽象

四个模块形成清晰的分层：

```
builtin_knowledge.py  ← 最底层：静态知识（不依赖运行时状态）
    ↑
function_summary.py  ← 中间层：静态分析结果（可持久化）
    ↑
trace_cache.py       ← 运行层：动态缓存（会话级生命周期）
    ↑
branch_constraint.py ← 语义层：程序语义增强（减少误报）
```

### 7.2 引擎无关性

四个模块不依赖任何特定语言的解析逻辑：
- `BranchConstraint` 只描述"变量 + 操作符 + 值"的抽象约束
- `FunctionSummary` 只记录"函数名 + 参数 + 返回值依赖"的抽象信息
- `TraceCache` 通过延迟加载桥接不同语言的知识库
- `BuiltinKnowledge` 通过语言名字典分离不同语言的知识

这种设计使得新增语言引擎时，共享模块**零修改**——只需创建新的语言子目录和实现语言专属的 `parser.py`（以及可选的 `builtin_knowledge.py`）。

### 7.3 容错设计

- **缓存失效安全**：TraceCache.clear() 在每次扫描开始时调用，避免脏数据
- **知识库降级**：如果语言知识库模块不存在，`_load_builtin_module` 返回 False，`lookup_builtin` 返回 None，引擎继续走函数体分析路径
- **摘要缺失处理**：如果函数没有摘要（摘要生成失败或函数定义不在扫描范围内），引擎直接进入函数体分析
- **缓存文件损坏**：SummaryCacheManager 对 JSON 解析失败做了 try-except 处理，损坏的缓存会被忽略并重新生成

### 7.4 数据驱动 vs 规则驱动

知识库是典型的**数据驱动**设计——新增一条知识只需添加一行字典条目，不需要修改任何分析逻辑。这使得知识库的扩展和维护成本极低，安全研究员可以快速添加新的函数知识而不需要理解引擎的内部实现。

规则引擎（rules/ 目录中的 YAML 规则）同样是数据驱动的设计——新增漏洞模式只需编写规则文件，不需要修改扫描引擎。这种"引擎 + 数据"的分离是 Kunlun-M 架构的核心优势。
